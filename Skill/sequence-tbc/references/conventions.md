# TBC Drawing Conventions — กติกาการวาด

กติกาที่ทำให้ไดอะแกรมทุกใบของบริษัท "หน้าตาเดียวกัน อ่านง่าย ใช้สื่อสารกับ Business ได้"

---

## 0. ⭐ 1 ไดอะแกรม = 2 ไฟล์ (กฎสำคัญที่สุด)
แยก "รูป" กับ "คำอธิบาย" ออกจากกันเสมอ — รูปต้องสะอาด พร้อมนำเสนอ
| ไฟล์ | เนื้อหา |
|---|---|
| `[ชื่อ].puml` | **เฉพาะรูป flow** — title, header, footer, ขั้นตอน, status note, alt |
| `[ชื่อ].md` | **คำอธิบายแยก** — ข้อมูลเอกสาร (เรื่อง/ระบบ/ขอบเขต) + Status Legend + สรุปช่วง + เงื่อนไข (ใช้ `templates/description-template.md`) |

> ❌ **ห้ามใส่** กล่อง "ข้อมูลเอกสาร" และ "Status Legend" ลงในรูป `.puml` (ทำให้รก) — ให้อยู่ในไฟล์ `.md` เท่านั้น

## 1. ภาษา
- **ไทยเป็นหลัก** สำหรับ flow / note / ชื่อขั้นตอน
- **คงคำเทคนิคเป็นอังกฤษ** (OTP, Token, API, POS, Pending, Approved)

## 2. Actor vs Participant
- `actor` = **คน** → `actor "User\n(ลูกค้า)" as User`
- `participant` = **ระบบ** → `participant "⚙️ CRM (Backend)" as CRM`
- **โลโก้ปิดเป็น default** ใช้ emoji/ข้อความแทน (🖥️ POS, 💬 LINE OA, ⚙️ CRM)
  เพราะการฝัง `<img:...>` ทำให้ไฟล์พังเวลาแชร์ (เคยเกิด `(Cannot decode)`)

## 3. ลูกศร
- `->` (เส้นทึบ) = ส่งคำขอ / สั่งงาน / action
- `-->` (เส้นประ) = ตอบกลับ / ผลลัพธ์
- `A -> A` = self-message (ระบบประมวลผลภายใน เช่น "ตรวจสอบสิทธิ์")

## 4. ⭐ Status ของ User (หัวใจของ TBC)
ทุกครั้งที่สถานะบนหน้าจอ User เปลี่ยน ให้ใส่ `note over User` พร้อม **สีตามความหมาย**:
```
note over User #D4EDDA
  📱 **สถานะ:** "อนุมัติแล้ว (Approved)"
  ⚡ **Trigger:** ส่ง LINE แจ้งเตือน + เปิดปุ่ม "รับสิทธิ์"
end note
```
- บรรทัด `📱 สถานะ:` = สิ่งที่ User **เห็นจริงบนจอ** (ไม่ใช่ค่าใน database)
- บรรทัด `⚡ Trigger:` = สิ่งที่ **ระบบทำต่อทันทีที่เป็นสถานะนี้** (ใส่เมื่อมี)

### สีมาตรฐาน
| สี | ความหมาย |
|---|---|
| `#FFF3CD` | รอ / กำลังดำเนินการ (Pending) |
| `#D4EDDA` | สำเร็จ / อนุมัติ (Success / Approved) |
| `#F8D7DA` | ผิดพลาด / ปฏิเสธ / หมดอายุ (Error / Rejected / Expired) |
| `#D1ECF1` | ข้อมูล / แจ้งเตือน (Info) |

> คำอธิบายสี (Status Legend) **ไม่ใส่ในรูป** — ไปอยู่ในไฟล์ `.md` แยก (ดูข้อ 0)

## 5. เงื่อนไขระบบ → `alt` / `else`
แสดงทางแยกของระบบ inline ในตัว flow (ไม่ยัดในหัวกระดาษ):
```
alt มีสิทธิ์คงเหลือ
    ...
else ไม่มีสิทธิ์คงเหลือ
    ...
end
```
ซ้อน `alt` ได้เมื่อมีเงื่อนไขย่อย (เช่น `[OTP valid]` ข้างใน `[First time login]`)

## 6. แบ่งช่วงด้วย `==`
```
== ส่วนที่ 1: ซื้อโควต้า ==
```

## 7. Metadata ในรูป (บังคับมีทุกใบ)
ใส่ใน `.puml` ได้เพราะเล็ก ไม่รก:
- `title` = **ชื่อ Flow** + คำอธิบายสั้น 1 บรรทัด
- `header` = `v[เวอร์ชัน] | [YYYY-MM-DD]`
- `footer` = `จัดทำโดย [ชื่อผู้ทำ] · [ทีม/บริษัท]`

ส่วน "ข้อมูลเอกสาร" แบบยาว (เรื่อง/ระบบ/ขอบเขต) → อยู่ในไฟล์ `.md` (ดูข้อ 0)

## 8. ตัวช่วยอ่านง่าย (เปิดใช้เสมอ)
- `autonumber "0."` — ใส่เลขลำดับ **มีจุด** ทุก step → `1. เปิดแอป`, `2. กรอก OA Number` (อ้างอิงตอนคุยกับ dev: "ขั้นที่ 7…")
- `activate` / `deactivate` — แถบ activation บน lifeline ที่กำลังทำงาน

## 9. ธีม
วางบล็อกธีมจาก `assets/tbc-theme.puml` **inline บนสุด** ของทุกไฟล์ — **ห้าม `!include`** (ให้ไฟล์ standalone)

## 10. ตั้งชื่อไฟล์
`[ระบบ]-[ชื่อ-flow].puml` + `.md` ชื่อเดียวกัน เช่น `mobile-login-flow.puml` คู่กับ `mobile-login-flow.md`

## 11. กันรูปโดนตัด
ไดอะแกรมยาว ๆ จะโดนตัดที่ 4096px — `scripts/render.sh` ใส่ `-DPLANTUML_LIMIT_SIZE=16384` ให้แล้ว
ถ้า render เองนอกสคริปต์ อย่าลืมใส่ flag นี้
